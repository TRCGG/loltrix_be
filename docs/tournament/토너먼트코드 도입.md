# 토너먼트 코드 도입 (리플 파싱 → Riot API 전환)

> 2026-07-06 논의 정리.
> **목표: 토너먼트 코드 발급을 위한 라이엇 프로덕션 키 신청용 dev MVP 만들기.**
> MVP는 개발 키 + `tournament-stub-v5`로 전체 플로우를 구현하고, 이를 근거로 운영키 + Tournaments API 접근을 신청한다.

---

## 1. 배경: 왜 전환하나

현재는 방장이 업로드한 리플 파일(.rofl)의 `statsJson`(게임 종료 시점 최종 스탯)만 파싱한다.
토너먼트 코드로 게임을 만들면 Riot Match-V5 API(`/lol/match/v5/matches/{matchId}` + `/timeline`)로 훨씬 풍부한 데이터를 자동 수집할 수 있다.

### 리플 파싱 vs Match-V5 데이터 비교

| 카테고리 | 리플(.rofl statsJson) | Match-V5 API |
|---|---|---|
| 밴픽 | ❌ 없음 | ✅ `teams[].bans[]` (밴 순서 포함) — 단 `pickType: TOURNAMENT_DRAFT`일 때만 |
| 경기 실제 시각 | ❌ 없음 → `played_date`를 업로드 시각으로 대체 중 (현재 최대 한계) | ✅ `gameCreation` / `gameStartTimestamp` / `gameEndTimestamp` |
| 챔피언 식별 | `SKIN` 영문명 → 매핑 실패 시 NULL 포기 | ✅ 숫자 `championId` + `championName` 직접 제공 |
| 최종 스탯 | raw 49개 + 자체 파생 14개 | ✅ 동등 이상 + `challenges` 블록 약 130개 (soloKills, killParticipation, 라인전 상대 비교 지표 등) |
| 딜 상세 | 총합만 | ✅ 물리/마법/고정 분리, 포탑딜, 최대 치명타 |
| 팀 오브젝트 선취 | ❌ (개인 킬 수만) | ✅ `teams[].objectives` — 바론/용/전령/유충/포탑별 first + kills |
| 퍼스트블러드/타워 | ❌ | ✅ 개인별 플래그 |
| 핑 | 4종 | ✅ 13종 |
| 스킬/스펠 사용 횟수 | ❌ | ✅ |
| **타임라인** | ❌ 전무 (최종 스냅샷 1장) | ✅ 1분 단위 프레임(좌표·골드·XP·CS·스탯) + 전체 이벤트 로그(킬 위치, 아이템 빌드 순서, 스킬 순서, 와드 위치, 오브젝트) |
| 리플 재생 | ✅ 원본 파일 보유 | ❌ (단, 토너먼트 코드 게임도 리플은 생성됨) |
| 수집 방식 | 방장 수동 업로드 | 게임 종료 시 자동 (콜백) |

타임라인으로 가능해지는 것: 골드 그래프, 15분 라인전 골드차(현 `lane_gold_diff`는 최종 골드 근사치), 동선/와드 히트맵, 빌드 순서 분석.

### 제약

- 코드 없이 만든 일반 커스텀 게임은 API로 소급 조회 **불가** → 리플 업로드와 **병행 운영** 전제.
- Tournaments API는 **프로덕션 키 전용**. 개발 키(24h 만료)로는 `tournament-stub-v5`(목업)만 가능.

## 2. 전체 아키텍처 플로우

```
[내전 방 개설]
  백엔드: POST /lol/tournament/v5/codes 로 코드 발급
    ├ pickType: TOURNAMENT_DRAFT (밴픽 데이터 확보), teamSize: 5, mapType: SUMMONERS_RIFT
    ├ metadata에 내부 ID 삽입: {"guildId":"...","customMatchId":"..."}
    └ DB 저장: tournament_code ↔ guild_id ↔ 내부 매치ID (status: PENDING)
  유저: 디스코드 봇이 채널에 코드 게시 → 각자 롤 클라이언트 [플레이]→[토너먼트 코드] 입력

[게임 종료]
  Riot → provider 등록 시 지정한 콜백 URL로 HTTP POST
    { shortCode, metaData, gameId, region, gameType, gameMap, ... }

[matchId 조립 및 수집]
  matchId = region + "_" + gameId   (예: "KR_1234567890")
  → GET /lol/match/v5/matches/{matchId}
  → GET /lol/match/v5/matches/{matchId}/timeline
  → 검증 후 저장, 코드 status: COMPLETED
```

### 발급 체인 (앞 2개는 사실상 1회성 설정)

1. `POST /providers` — 지역(KR) + 콜백 URL 등록 → providerId. **키 재발급 시 provider 재등록 필요.**
2. `POST /tournaments` — providerId → tournamentId.
3. `POST /codes?tournamentId=...` — 판마다 발급. 코드에 로비 규칙이 박혀 유저가 변경 불가.

### 코드는 1회용 (판마다 새 코드 + 새 로비)

- 공식 가이드상 **코드 하나 = 게임 한 판**. 재사용하면 콜백이 판마다 제대로 오지 않음.
- 기존 커스텀은 게임 후 같은 방에서 재경기가 가능했지만, 코드 방식은 다음 판마다 전원이 새 코드로 재입장해야 함 → 연속 내전에서 체감되는 유일한 추가 마찰.
- 완화: `count` 파라미터로 **한 번에 최대 1,000개 발급 가능** → `/내전시작` 시 코드 여러 개를 미리 발급해두고, 콜백 수신(판 종료) 시 봇이 다음 코드를 자동 게시.
- 대신 판 종료 후 **리플 업로드 단계가 사라지므로** 전체 마찰 총량은 오히려 감소 (업로드 누락으로 인한 기록 유실도 제거).

### 콜백 유실 대비 (폴백)

- 주기 폴링: `GET /lol/tournament/v5/games/by-code/{code}` — PENDING 상태로 N시간 지난 코드 조회.
- 최후 수단: 참가자 puuid로 `GET /matches/by-puuid/{puuid}/ids?type=tourney` → 각 매치의 `info.tournamentCode`를 DB 코드와 대조.
- ⚠️ Match-V5에는 v4에 있던 `by-tournament-code/ids` 엔드포인트가 없다.

### 보안 주의

- **콜백 엔드포인트는 무인증** — 아무나 가짜 POST 가능. 대응:
  1. 콜백 URL에 추측 불가능한 시크릿 경로 사용
  2. 콜백은 **트리거로만** 사용 — 받은 gameId로 match-v5를 실제 호출해 `info.tournamentCode`가 DB의 우리 코드와 일치할 때만 저장
- API 키는 서버 환경변수로만 관리, 노출 시 즉시 재발급 (provider도 재등록됨에 유의).

## 3. 프로덕션 키 발급 요건 및 리스크

### 심사 요건

| 요건 | 현황 |
|---|---|
| 작동하는 서비스/프로토타입 | ✅ 실서비스 운영 중 (리플 기반) |
| 서비스 소개 공개 웹사이트 + 유저 플로우 설명 | ⚠️ 정리 필요 — "검증 가능한 웹사이트 없이는 발급 불가" 명시 |
| Tournaments API 사용 능력 입증 | ❌ → **이 문서의 MVP가 해당** (stub로 전체 플로우 구현) |
| 콜백 수신용 공개 HTTPS 서버 | ⚠️ CD 마무리 필요 |
| 무료 티어 | ✅ 무료 서비스 |
| 심사 기간 | 약 20영업일+ |

### 리스크

- **"내전"이 Tournament API 용도로 인정되는가**가 최대 변수. 정책상 명목은 토너먼트 운영(참가자 20인 이상 등 조항 존재). 다만 상금·참가비·도박 요소가 없고, 내전/스크림 도구가 승인된 커뮤니티 사례 존재.
  → 신청서에 용도를 포장하지 말고 **"guild in-house custom game stat tracking"**으로 정직하게 기재 (용도 불일치 적발 시 키 회수).
- 서비스명 "loltrix"의 "lol" 접두어가 네이밍 정책에서 지적될 가능성 (케이스바이케이스).

## 4. MVP 작업 목록 (개발 키 + tournament-stub-v5)

stub는 실제 클라이언트에서 코드가 작동하지 않을 뿐, API 구조는 동일 → 코드를 미리 전부 만들어둘 수 있다.

- [ ] **Phase 1 — 발급 플로우**
  - [ ] Riot API 클라이언트 모듈 (키 관리, rate limit 대응)
  - [ ] provider / tournament 등록 (stub)
  - [ ] 코드 발급 API: `POST /codes` 호출 + `tournament_code` 테이블 저장 (code, guild_id, custom_match_id, metadata, status)
  - [ ] 디스코드 봇(또는 웹) 연동: 내전 생성 → 코드 발급 → 채널 게시
- [ ] **Phase 2 — 콜백 수신**
  - [ ] 시크릿 경로 콜백 엔드포인트 (`POST /callback/riot/{secret}`)
  - [ ] shortCode → DB 대조 → matchId 조립 → match-v5 검증 호출 → status 갱신
  - [ ] 폴백 폴링 잡 (`games/by-code`, PENDING 코드 대상)
  - [ ] 연속 내전 지원: 콜백 수신(판 종료) 시 봇이 미리 발급해둔 다음 코드를 자동 게시
- [ ] **Phase 3 — 데이터 적재**
  - [ ] match-v5 응답 → 기존 `match_participant` / `mmr_participant_metric` 스키마 매핑 (리플 파싱 결과와 동일 형태로 정규화)
  - [ ] `played_date`를 `gameStartTimestamp`로 대체 (리플 경로는 기존 업로드 시각 유지)
  - [ ] 밴픽 저장 스키마 신설 (`teams[].bans`)
  - [ ] (후순위) timeline 적재 설계
- [ ] **Phase 4 — 신청 준비**
  - [ ] 서비스 소개 페이지 정리 (유저 플로우 스크린샷 포함)
  - [ ] CD 마무리 → 콜백 서버 공개 HTTPS 배포
  - [ ] 개발자 포털 앱 등록 → 운영키 + Tournaments API 접근 신청 (MVP 시연 자료 첨부)

승인 대기 중에도 리플 업로드 방식은 그대로 운영한다 (병행 전제, 서비스 중단 리스크 없음).
