## 배경 / 문제

토너먼트 코드 전환 기획(2026-07-06)을 실행에 옮기기 전에, 기획 자체를
전방위(승인 리스크·노력 대비 효익·유저 마찰·데이터 활용처)로 레드팀 검토한다.
산출물은 **Go / No-Go / 조건부 Go 권고 + 근거**.

## 목표 / 비목표

**목표**
- 킬 조건 판정: **내전/스크림/커스텀 게임 기록 서비스가 Tournaments API가
  포함된 프로덕션 키를 실제 승인받은 사례를 최소 1건, 검증 가능한 출처와 함께 확인**
  - 사례 확인 = Go 필요조건. 미확인 = No-Go(보류).
- 나머지 3개 관점(공수, 유저 마찰, 데이터 활용처)은 반론 → 문서의 대응 존재 여부
  → 잔여 리스크 순으로 평가 (킬 조건 아님, 감수 가능 여부 판단용)

**비목표**
- Phase 1~4 구현 명세 작성 (Go 판정 이후의 일)
- 기획 문서 자체의 수정 (재검토 결과 반영은 별도 결정)
- 대안 아키텍처 설계

## 결론 (2026-07-06 재검토 완료)

### 판정: **No-Go (보류)** — 킬 조건 미충족 + 파이프라인 자체가 현재 고장

### 킬 조건 판정 근거

**승인 사례: 0건.** 상금 없는 내전/스크림 기록 서비스가 Tournaments API 포함
프로덕션 키를 승인받은 검증 가능한 사례를 찾지 못했다. 오히려 반대 방향 증거가 나왔다.

| # | 증거 | 등급 | 출처 |
|---|---|---|---|
| 1 | **정확히 같은 용도의 부정 사례**: 내전용 토너먼트 코드 발급 봇이 "Not deployable as riot tournament api policy does not support creating single games"라고 README에 명시 | 원문 직접 확인 | [artemigkh/league-of-legends-inhouse-organizer-discord-bot](https://github.com/artemigkh/league-of-legends-inhouse-organizer-discord-bot) |
| 2 | **정책상 토너먼트 최소 요건 "Must have at least 20 participants"** — 10인 내전 로비는 라이엇 정의상 '토너먼트'에 미달 | 원문 직접 확인 | [General Policies](https://developer.riotgames.com/policies/general) |
| 3 | **치명적**: 토너먼트 코드로 만든 경기가 2026-04-29부터 Match-V5에서 404 반환. 콜백·매치ID는 정상 수신되나 조회 불가. 여러 커뮤니티 토너먼트 운영자가 영향. 라이엇 공식 응답 없음, 2026-07-06 현재 Open | 원문 직접 확인 | [developer-relations#1156](https://github.com/RiotGames/developer-relations/issues/1156) |
| 4 | Tournament-V5 접근은 승인된 프로덕션 앱 전제 (신청 경로 자체는 존재) | 워크플로우 3-0 검증 | [Tournament V5 공지](https://www.riotgames.com/en/DevRel/tournament-v5-is-coming) |
| 5 | 주요 내전 도구들이 하나같이 Tournaments API를 안 씀: InHouseQueue는 Riot API 자체를 미연동(/win 투표로 수동 기록), LoL Scrims Finder는 stub 플래그만 있고 프로덕션 키 증거 없음 + 결과는 스크린샷 수동 기록 | 리서치 수집 (미검증) | [InHouseQueue docs](https://docs.inhousequeue.xyz/docs), [LoL-scrims-finder](https://github.com/DannyMichaels/LoL-scrims-finder) |
| 6 | 간접 긍정: #1156에 커뮤니티 토너먼트 운영자들이 다수 등장 → "커뮤니티 토너먼트" 용도 승인은 실존. 단 이는 진짜 토너먼트 운영이지 내전 상시 기록이 아님 | 리서치 수집 (미검증) | 동일 이슈 스레드 |

**해석**: 증거 1·2가 "내전 = 토너먼트 아님"이라는 정책 해석을 뒷받침하고, 이 상태에서
"guild in-house custom game stat tracking"으로 정직하게 신청하면 용도 미달로 거절될
가능성이 높다. 증거 6처럼 진짜 토너먼트 운영자로는 승인 사례가 있으나, 그건 우리 용도가 아니다.

**증거 3은 킬 조건과 무관하게 단독으로도 보류 사유다** — 설령 키를 받아도
"코드 발급 → Match-V5 수집" 파이프라인이 2개월째 고장·방치 상태다. 기획서의 핵심 가치
(밴픽·타임라인·자동 수집)가 지금은 승인 여부와 무관하게 실현 불가능하다.

### 비-킬 관점 평가 (참고용, 코드베이스 실사 기반)

- **공수 (기획서 추정보다 큼)**: Riot API 호출 코드가 전무해 키 관리·rate limit·리전
  라우팅 전부 신규. 기존 statsJson→DB 매핑(matchParticipant.service, mmrMetric.service)은
  리플 포맷 강결합이라 Match-V5용 어댑터 별도 필요. 스케줄러 인프라 없음(폴백 폴링은
  스케줄러 도입부터). 승인 실패 시 이 공수 대부분이 매몰된다.
- **CD**: 기획서의 "CD 마무리 필요"는 과대평가 — CI/CD·무중단 배포는 이미 동작.
  실제 부족분은 콜백용 공개 HTTPS 노출 설정과 마이그레이션 자동화뿐.
- **유저 마찰 / 데이터 활용처**: No-Go 판정으로 이번엔 판단 불요. 재개 시 재평가.

2. 내전/스크림 기록 용도의 승인 사례가 새로 확인되거나, 라이엇이 해당 용도 자격을
   공식 확인 (MVP를 만들기 전에 developer portal 문의/신청서로 저비용 타진 가능)
