## 배경 / 문제

토너먼트 코드 전환 기획(2026-07-06)을 실행에 옮기기 전에, 기획 자체를
전방위(승인 리스크·노력 대비 효익·유저 마찰·데이터 활용처)로 레드팀 검토한다.
산출물은 **Go / No-Go / 조건부 Go 권고 + 근거**.

## 목표 / 비목표

**목표**
- 킬 조건 판정: **내전/스크림/커스텀 게임 기록 서비스가 Tournaments API가
  포함된 프로덕션 키를 실제 승인받은 사례를 최소 1건, 검증 가능한 출처와 함께 확인**
  - 사례 확인 = Go 필요조건. 미확인 = No-Go(보류).

  → 잔여 리스크 순으로 평가 (킬 조건 아님, 감수 가능 여부 판단용)

**비목표**

- 대안 아키텍처 설계

### 최종 판정: **Go** (2026-07-07 번복)

2026-07-06 최초 판정은 No-Go(보류)였으나, **ARRA.gg 반례가 확인되어 킬 조건이
충족**됨에 따라 합의된 판정 규칙("사례 확인 시 나머지 걱정은 감수하고 Go")대로 번복한다.

#### 번복 근거: ARRA.gg (2026-07-07 확인)

- ARRA.gg는 한국의 LoL **내전·스크림 매칭 플랫폼**(2026-03 베타 오픈, 무료,
  "isn't endorsed by Riot Games" 고지 = 서드파티). https://arra.gg
- 팀원이 **2026년 5월 이후** ARRA.gg 내전 세션에서 토너먼트 코드를 발급받아
  롤 클라이언트로 입장, 경기 후 **승패·KDA가 자동 기록**되는 것을 직접 확인.
  - stub 코드는 실제 클라이언트에서 작동하지 않고, Tournaments API는 프로덕션 키
    전용이므로 → **상금 없는 내전 플랫폼이 프로덕션 Tournaments API를 승인받아
    운영 중인 실증 사례**. 킬 조건 충족.
  - 증거 등급: 팀원 직접 사용 경험 (공개 문서로는 미확인 — ARRA 공개 페이지·공지에는
    토너먼트 코드 언급 없음, 로그인 뒤 세션 화면에서만 노출되는 것으로 보임)
- 부수 효과: 최초 판정의 부정 증거들이 약화됨 —
  - artemigkh 부정 사례(2019)는 노후 정보였음이 확인됨
  - "참가자 20인 이상" 정책 조항은 실무 심사에서 내전 용도를 막지 않음이 실증됨
  - **#1156(Match-V5 404)도 KR 내전 플로우에서는 실제로 작동함이 실증됨**
    (5월 이후 사용인데 자동 기록 성공. 단 이슈 자체는 2026-07-07 현재 여전히 Open,
    라이엇 응답 없음, 마지막 활동 5/30 — 특정 리전/키에 국한된 문제이거나 조용히
    일부 복구된 것으로 추정. 코멘트에 RSO 토큰 경유 조회가 대안으로 언급됨)

#### Go 이후 유의사항

1. **#1156 추적 유지** — 우리 키에서도 Match-V5 조회가 되는지 Phase 2(콜백 수신)
   구현 시 가장 먼저 실증할 것. 404 발생 시 RSO 토큰 경유 조회를 폴백 후보로 검토.
2. **신청서에 ARRA.gg 유사 사례를 참조점으로 활용 가능** — 같은 시장(KR 내전)에서

### 킬 조건 판정 근거

**승인 사례: 0건.** 상금 없는 내전/스크림 기록 서비스가 Tournaments API 포함

| # | 증거 | 등급 | 출처 |

**해석**: 증거 1·2가 "내전 = 토너먼트 아님"이라는 정책 해석을 뒷받침하고, 이 상태에서
"guild in-house custom game stat tracking"으로 정직하게 신청하면 용도 미달로 거절될
가능성이 높다. 증거 6처럼 진짜 토너먼트 운영자로는 승인 사례가 있으나, 그건 우리 용도가 아니다.

**증거 3은 킬 조건과 무관하게 단독으로도 보류 사유다** — 설령 키를 받아도
"코드 발급 → Match-V5 수집" 파이프라인이 2개월째 고장·방치 상태다. 기획서의 핵심 가치
(밴픽·타임라인·자동 수집)가 지금은 승인 여부와 무관하게 실현 불가능하다.

### 비-킬 관점 평가 (참고용, 코드베이스 실사 기반)

- **공수 (기획서 추정보다 큼)**: Riot API 호출 코드가 전무해 키 관리·rate limit·리전
  라우팅 전부 신규. 기존 statsJson→DB 매핑(matchParticipant.service, mmrMetric.service)은
  리플 포맷 강결합이라 Match-V5용 어댑터 별도 필요. 스케줄러 인프라 없음(폴백 폴링은
  스케줄러 도입부터). 승인 실패 시 이 공수 대부분이 매몰된다.
- **CD**: 기획서의 "CD 마무리 필요"는 과대평가 — CI/CD·무중단 배포는 이미 동작.
  실제 부족분은 콜백용 공개 HTTPS 노출 설정과 마이그레이션 자동화뿐.
- **유저 마찰 / 데이터 활용처**: No-Go 판정으로 이번엔 판단 불요. 재개 시 재평가.
