# TRC-230 대회 기능 — 백엔드 (대회·팀·로스터 API + 경기 연결·순위 집계)

> 티켓: [TRC-230](https://trcgg.atlassian.net/browse/TRC-230) (백엔드 파트)
> 공통 우산: [TRC-229](https://trcgg.atlassian.net/browse/TRC-229) `[공통] 대회 기능 도입`
> 프론트 파트: TRC-231 (`front/trcgg_front/docs/plans/TRC-231-대회기능-프론트.md`)
> 봇 파트: TRC-232 (`bot/trcgg_bot/docs/plans/TRC-232-대회기능-봇.md`)

## 배경 / 문제

일반내전만 있는 서비스에 **대회** 도메인을 추가한다. AI 인터뷰(2026-07-08)로 확정된 명세:

- 대회 = guild_manager가 개설하는 경기 기록의 묶음. **형식(풀리그/브래킷) 엔진 없음**
  — 클랜마다 진행 방식이 달라 강제 불가 (인터뷰 결정)
- 경기는 토너먼트코드로 진행 — TRC-225 MVP 인프라(발급 체인·콜백·Match-V5 적재·폴링) 재사용
- 스크림/본경기 2분류. 대회별 분리 전적 + 승수(동률 시 승률) 자동 집계 순위표
- **대회 경기는 일반 전적·통계·MMR에서 완전 제외** (인터뷰 결정: 완전 분리)

인터뷰 후속 확정 (2026-07-08 기획 검토):

- 라이프사이클 **3단계 준비(DRAFT)→진행(OPEN)→종료(CLOSED)** — 준비 중 발급 불가
- 순위표는 **스크림/본경기 각각** 집계
- 웹 발급 코드는 **화면 표시만** (봇 채널 게시 없음)
- 종료 잠금 범위: **발급·팀/로스터 수정만 차단, 미분류 보정은 종료 후에도 허용**

기존 의도(intent-guard 확인):

- TRC-225 적재 파사드(`tournamentSave.facade.ts`)는 리플 경로와 동일 스키마
  (`custom_match`/`match_participant`/`mmr_participant_metric`)에 적재 — 이 위에
  대회 태깅을 얹되 **리플 경로(`replaySave.facade.ts`)는 수정 금지** (TRC-225 불변식)
- `tournament_code.metadata`(jsonb)는 채널 ID 등 발급 컨텍스트 저장용으로 설계됨
  (커밋 a18f628) — 대회·구분 정보도 여기가 아닌 **정식 컬럼**으로 둘지 단계 1에서 확정
- TRC-222(커밋 9ff7878)로 guildManager의 웹 권한 부여 API 체계 존재 — 개설 권한 재사용

## 선행 개발 (dev 머지 필수 — 착수 게이트)

본 계획은 **dev 기준**으로 진행한다 (2026-07-08 결정). 아래 기능들은 현재
**피처 브랜치에만 존재하며 dev에 머지되지 않았다** (확정 기능 아님, 테스트 단계).
전부 dev에 머지된 뒤에만 본 작업을 착수한다 — 테스트 단계 브랜치 위에 쌓으면
선행 기능이 변경·폐기될 때 재작업이 된다.

| 선행 | 제공하는 것 (본 계획이 의존) | 상태 · 남은 게이트 |
|---|---|---|
| **TRC-225** 토너먼트코드 MVP | `tournament`/`tournament_code` 스키마(마이그레이션 008), Riot 클라이언트, 코드 발급 API, 콜백 수신·재검증, Match-V5 어댑터, `tournamentSave.facade`(본 계획의 적재 훅을 여기 확장), 폴링 잡 | 브랜치 `TRC-225-Back-토너먼트코드-MVP`. **머지 금지(2026-07-07)** — 프로덕션 키 승인 + 실경기 검증(TRC-225 단계 8) 통과 후 dev 머지 |

봇 파트 선행(TRC-226)은 봇 계획 문서 참조. 프론트(TRC-231)는 본 티켓의 API
계약 확정이 선행이므로 자동으로 이 체인 뒤에 선다.

## 목표 / 비목표

**목표**

1. 대회·팀·로스터 스키마 + 라이프사이클(개설→진행→종료, 종료 시 잠금)
2. guild_manager 전용 관리 API (개설·종료·팀/로스터 CRUD·웹 코드 발급)
3. 코드 발급 확장: 대회 ID + 구분(스크림/본경기) 지정
4. 적재 훅: 로스터 대조 팀 자동 매칭 (불일치 시 미분류 보존)
5. 대회 조회 API: 구분별 분리 전적, **스크림/본경기 각각의** 승수 순위표, 미분류 목록·수동 보정
6. **분리 보장**: 일반 전적·통계·MMR 쿼리에서 대회 경기 제외 (누수 점검 포함)

**비목표**

- 대진표/일정, 밴픽 통계 API, 최종 순위 수동 확정, 형식 엔진 (다음 버전)
- 리플 업로드 경로 변경 — 대회 경기는 토너먼트코드로만 진행

## 변경 범위 (파일·API·스키마) — 잠정, 단계 1 실사 후 확정

### 스키마 (Drizzle + 수기 SQL `migrations/009_competition.sql`)

| 테이블 | 내용 |
|---|---|
| `competition` | guild_id, name, status(`DRAFT`준비/`OPEN`진행/`CLOSED`종료), 개설자, 개설/종료 시각. **준비 상태에서는 코드 발급 불가**(팀/로스터 등록 기간) |
| `competition_team` | competition_id, name |
| `competition_roster` | team_id, riot 계정 식별자(puuid 또는 riot_account 참조 — 실사 후 확정) |
| `competition_match` | custom_match_id(unique), competition_id, match_type(`SCRIM`/`MAIN`), team_a_id·team_b_id·winner_team_id(nullable — 미분류 허용) |
| `tournament_code` 확장 | `competition_id`(nullable FK), `match_type`(nullable) 컬럼 추가 — null이면 기존 일반내전 코드(TRC-226 경로) 그대로 |

**분리 태깅(잠정안)**: `competition_match`에 행이 존재하는 `custom_match`가 대회 경기.
일반 전적·통계·MMR 쿼리에 `NOT EXISTS (competition_match)` 필터 추가 — 또는
`custom_match`에 nullable `competition_id` 직접 태깅. **두 안 중 기존 쿼리 수정
범위가 작은 쪽을 단계 1 실사에서 확정** (쿼리 누수가 수용 기준이므로 전수 조사 필수).

### API (라우트 배치는 TRC-225 규칙 준수 — 봇 전용/웹 인증 구역 구분)

| 구역 | 엔드포인트(잠정) | 내용 |
|---|---|---|
| 웹(guild_manager) | `POST/PATCH /competitions` | 개설·종료(잠금) |
| 웹(guild_manager) | `POST/PUT/DELETE /competitions/:id/teams`, `…/teams/:tid/roster` | 팀·로스터 관리 |
| 웹(guild_manager) | `POST /competitions/:id/codes` | 웹 코드 발급 (구분 지정, 코드 응답으로 반환) |
| 웹(guild_manager) | `PATCH /competitions/:id/matches/:mid` | 미분류 경기 수동 보정 (팀·구분 지정) |
| 웹(공개) | `GET /competitions`, `GET /competitions/:id` (전적·순위표·미분류 포함) | 조회 |
| 봇 전용 | 기존 `POST /tournament/codes` 확장 | competitionId·matchType 파라미터 추가 |

### 서비스/파사드

- `competition.service.ts` 신설 (CRUD·라이프사이클·순위 집계)
- `tournamentSave.facade.ts` 확장: 적재 트랜잭션 안에서 코드의 competition_id를 보고
  `competition_match` 행 생성 + 로스터 대조 팀 자동 매칭. **기존 트랜잭션 순서 불변.
  MMR 제외는 확정(완전 분리) — 제외 방식(적재 생략 vs 조회 필터)만 분리 태깅안과 함께 확정**
- 권한: TRC-222의 guildManager 검증 미들웨어/패턴 재사용

## 영향받는 불변식 / 리스크

- **리플 경로 불변** (TRC-225): `replaySave.facade.ts`/`replay.service.ts` 수정 금지
- **콜백 무인증 방어 불변** (TRC-225): match-v5 재검증 후에만 적재 — 대회 태깅도 이 뒤에서만
- **tournamentSave 트랜잭션**: markCompleted가 tx 안에 있는 구조(단계 4 ⚠️ 요건) 유지 —
  competition_match 생성 실패 시 PENDING 유지되도록 같은 tx에 포함
- 분리 필터 누수 = 조용한 데이터 오염: 단계 1 전수 조사 없이 필터를 붙이면
  일부 통계에 대회 경기가 새어 들어가도 눈치채기 어려움 → 단계 7을 독립 단계로 명시
- 로스터 자동 매칭 오판(용병·닉변): 강제 배정하지 않고 미분류 보존이 원칙 —
  매칭 임계(예: 로스터 일치 5인 중 몇 명 이상)는 단계 5에서 확정
- **선행 기능이 아직 테스트 단계**: TRC-225/222 산출물의 인터페이스가 dev 머지
  전까지 바뀔 수 있음 — 본 문서의 '변경 범위'는 잠정이며 단계 0에서 재검토
